# Issue調査レポート作成

GitHub issueまたはトピックを起点に、コードベースを調査し構造化されたレポートを出力する。

## 目的
- issueの問題が **なぜ起きているのか** をコードレベルで特定する
- 関連する **専門用語を初学者にもわかるように解説** する
- **公式ドキュメントや信頼できる記事** をリンク+引用テキスト付きで提示する
- 修正の方向性と期待される効果を明確にする

## 手順

### 1. Issue内容の把握
- issue番号が渡された場合: `gh issue view <番号> --repo yuuttana1223/web-speed-hackathon-2026` で内容を取得
- トピックが渡された場合: そのトピックに関連するissueがないか確認

### 2. コードベース調査
- issueに記載された問題箇所を `Glob` / `Grep` / `Read` で特定
- 問題の根本原因となるコード断片を引用（ファイルパス:行番号つき）
- **なぜそこにあたりをつけたのか**（調査の思考過程）を必ず記述する

### 3. 調査レポート出力

以下のフォーマットで会話にレポートを出力する:

```markdown
# 調査レポート: [issue タイトル]

## 概要
[1-2文で問題の要約]

## 調査の着眼点
[なぜこの箇所を調べたのか、どういうロジックであたりをつけたのかを説明]

## 問題の詳細

### 問題箇所 1: [小見出し]
**ファイル**: `path/to/file.ts:行番号`
**コード**:
```ts
// 問題のあるコード断片
```

**なぜこれが問題か**:
[パフォーマンスへの影響を具体的に説明]

**影響するスコア指標**: [TBT / LCP / CLS / FCP / SI のうち該当するもの]

---

## 用語解説

| 用語 | 解説 |
|------|------|
| 用語1 | わかりやすい説明 |
| 用語2 | わかりやすい説明 |

## 参考資料

### [記事/ドキュメントタイトル](URL)
> 引用テキスト（該当箇所の原文を引用）

要約や補足説明

---

## 修正の方向性
- [ ] 修正タスク1（期待効果: ○○）
- [ ] 修正タスク2（期待効果: ○○）

## 期待されるスコア改善
| 指標 | 改善見込み | 根拠 |
|------|-----------|------|
| TBT | ○○ | ○○ |
```

## 調査時の原則
- **推測ではなくコードで裏付ける**: 「おそらく」ではなく、該当コードを実際に見つけて引用する
- **WebSearchを積極活用**: 公式ドキュメントのURLや最新のベストプラクティスを検索して引用する
- **スコアリング重みを意識する**: TBT(x30) > LCP(x25) = CLS(x25) > SI(x10) = FCP(x10)
